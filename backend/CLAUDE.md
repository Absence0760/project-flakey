# backend

Express + Node + TypeScript API. Multi-tenant via Postgres Row-Level Security.

## Commands

- `npm run dev` — tsx watch, loading the committed `.env.development` then the gitignored `.env.development.local` (overrides). `seed`/`test` deliberately load **no** env file so they run on code defaults — that's how seed gets the `flakey` superuser while the app runs as non-superuser `flakey_app`; don't add `--env-file` to them.
- `npm run seed` — seed sample data: 3 primary users (`admin@example.com`/`admin`, `demo@example.com`/`demo123`, `viewer@example.com`/`viewer123`), 2 primary orgs (Acme Corp, Demo Team), ~85 runs across ~25 suites spanning Mochawesome/Playwright/JUnit, plus demoable extras: a known admin API key (`fk_demoadmindemoadmindemoadmindemoa`) for CLI testing, one webhook target at `https://example.invalid/seeded-hook`, two quarantine entries, two non-default error-group statuses (investigating + known), and one pending org invite (`token: demo-invite-token-do-not-use-in-prod-aaaa`). Also seeds **4 worker tenants** (`admin+w{0..3}@example.com` / `worker{0..3}123`, orgs `acme-w{0..3}`) with the same playground content as Acme (minus the global-unique API key + invite token) so parallel Playwright workers can each operate on a dedicated tenant. Override the worker-tenant count with `E2E_WORKER_TENANTS=N` (default 4; set to 0 for a slim production seed).
- `npm test` — `node --test` over `src/tests/**/*.test.ts` via tsx
- `npm run build` — `tsc` → `dist/`
- `npm start` — run the built `dist/index.js`
- `npm run rotate-keys` — re-encrypt all org secrets under the current primary key (see `docs/integrations.md` for the dual-key rotation procedure)
- `npm run replay-payload -- <path> [--reporter <type>] [--pretty]` — feed a captured reporter payload (Cypress/mochawesome JSON, Playwright/Jest/WebdriverIO JSON, JUnit XML) straight through the normalizer and dump the `NormalizedRun` to stdout (stats summary on stderr). No DB, no auth — a sub-second loop on ingestion bugs without standing up the stack. Reporter is auto-detected from the filename when `--reporter` is omitted.

## Package manager

Use **npm** here, not pnpm. The backend has its own lockfile and is intentionally outside the pnpm workspace.

## Layout

- `src/integrations/` — Jira, PagerDuty, coverage-gate logic; scheduled reports live in `src/scheduled-reports.ts`
- `src/git-providers/` — GitHub/GitLab/Bitbucket PR-comment + commit-status adapters. GitHub also posts per-failure inline diff annotations via the Checks API (`postChecksAnnotations` in `github.ts`; annotation derivation in `annotations.ts`) — needs a `checks:write` token. All three also implement repo-**write** primitives (`getDefaultBranch`/`getFileContent`/`createBranch`/`commitFile`/`createPullRequest`) used by the AI fix-PR flow — those need an elevated WRITE/PR token (see Key constraints). See `docs/integrations.md`.
- `src/auto-quarantine.ts` — deterministic auto-quarantine evaluated at run finalization (see Key constraints).
- `src/flaky-analysis.ts` — the shared windowed flaky detector (`computeFlakyTests`) used by the `/flaky` route, auto-quarantine, and the flaky webhook events.

Everything else (`src/routes/`, `src/normalizers/`, `src/tests/`) is self-describing.

## Ports

- API listens on `PORT` (default `3000`). Health probe: `GET /health`.
- Dev frontend defaults to `7778` (Vite); `CORS_ORIGINS` default in dev allow-lists `http://localhost:7778,http://localhost:3000`.

## Key constraints

- Runs as DB user `flakey_app` (non-superuser) so RLS policies apply. Don't bypass this by connecting as a superuser.
- Migrations live in `migrations/`; apply via `./migrate.sh`.
- `JWT_SECRET` is required in production (no default).
- **First admin is env-gated — no default credentials ship.** A fresh DB comes up with zero users (migration `003_auth.sql` no longer seeds a known admin). To create the first admin, set both `FLAKEY_BOOTSTRAP_ADMIN_EMAIL` and `FLAKEY_BOOTSTRAP_ADMIN_PASSWORD`; on boot, `src/bootstrap-admin.ts` idempotently creates that user (role `admin`, bcrypt cost 12), a personal org, and an `owner` membership — mirroring `POST /auth/register`. If either var is unset it no-ops. It never resets an existing user's password, so it's safe to leave set across restarts. In dev, `npm run seed` still creates `admin@example.com`/`admin` directly (independent of this bootstrap).
- Secrets for integrations (Jira tokens, PagerDuty keys) are AES-256-GCM encrypted via `FLAKEY_ENCRYPTION_KEY`. If the key is unset, the code falls back to plaintext passthrough — only acceptable in local dev. The key FORMAT (32 bytes, hex or base64) is validated at boot — a malformed value refuses to start, not just an unset one.
- `SMTP_*` + `EMAIL_FROM` control transactional mail (auth verification, password reset) and scheduled report delivery; all have safe defaults for local dev.
- Login hardening (see `docs/architecture.md` § 4 for the full picture): per-account lockout via `LOGIN_LOCKOUT_THRESHOLD` (default 5) / `LOGIN_LOCKOUT_MINUTES` (default 15) on top of the per-IP `AUTH_RATE_LIMIT_MAX` gate, bcrypt-bounded response time on unknown emails (no enumeration via timing), refresh-token revocation + rotation via the `revoked_refresh_tokens` table, and per-request org-membership re-validation in `requireAuth` so removed members 401 immediately.
- **Enterprise SSO is flag-gated and OFF by default.** `FLAKEY_SSO_ENABLED=true` exposes the OIDC login flow (`/auth/sso/*`) + admin config API (`/sso/config`); `PUBLIC_API_URL` sets the IdP redirect base (`<PUBLIC_API_URL>/auth/sso/callback`). SSO is **additive** — the callback mints the *existing* Flakey JWT; nothing changes post-login (same `requireAuth`, `org_members` re-check, RLS). IdP client secrets are encrypted via the same `FLAKEY_ENCRYPTION_KEY` path; role claims can only reach an admin-configured `role_map` value. It is a GovRAMP-scoped auth control — do not enable in a regulated env without CISO sign-off. See `docs/sso.md`. OIDC + SAML login + SCIM provisioning (Slices 1–3) are all built. SAML uses `@node-saml/node-saml` (vetted lib — don't hand-roll XML signatures) with InResponseTo binding + one-time-assertion replay tables. SCIM (`/scim/v2`, per-org bearer token via `POST /sso/scim/token`) maps create/deactivate/delete to org membership — deactivation removes the `org_members` row so the existing per-request re-check revokes access; resources are RLS-isolated per org.
- **Cross-org support access is `is_support` + read-only.** `users.is_support` (migration 053) is a platform flag set out-of-band by an operator — there is no API to grant it. A support user calls `POST /support/orgs/:orgId/token` to mint a 30-min read-only "view as org" JWT (audited as `support.session.start` in the target org). `requireAuth` clamps such a session to `GET` on an allow-listed read surface (see `SUPPORT_READ_BASEURLS` in `auth.ts`) — never the secrets/config routes — and re-checks `is_support` live so revoking it ends sessions immediately. No user impersonation. Details in `docs/architecture.md` § 4.
- **Audit log is hash-chained + exportable (SOC 2 / GovRAMP logging controls).** `logAudit` (`src/audit.ts`) appends every mutation into a per-org SHA-256 hash chain: each row binds the previous row's `entry_hash`, serialized by a transaction-scoped per-org advisory lock so the chain has one head (`src/audit-chain.ts`). `GET /audit/verify` (admin+) walks it and reports the first broken link — tamper-evidence for edits/deletes/reorders. Rows predating migration `064` are a counted legacy prefix (no backfill — would lock prod `audit_log`). **Export** (`src/audit-export.ts`) streams audit rows to a per-org HTTP (customer SIEM) or S3 destination (`audit_export_config`, RLS), durable + gap-free + at-least-once via a per-destination `id` cursor (gap-free *because* the per-org append lock makes that org's ids commit in increasing order — the two features share the lock). Off by default behind `FLAKEY_AUDIT_EXPORT_ENABLED` (flusher no-ops + `/audit/export` routes 404 when unset); the auth token is `crypto.ts`-encrypted and never returned; the HTTP target goes through the webhook SSRF guard at send time; delivery errors are sanitized (no upstream body/URL/token). GovRAMP-scoped — don't enable in a regulated env without CISO sign-off. See `docs/audit-logging.md`. **`logAudit` is now a transaction + advisory lock, so await it on hot paths** (don't fire-and-forget) or concurrent same-org callers pile up connections + lock waits.
- **Auto-quarantine is opt-in, deterministic, and never clobbers a manual entry.** When an org sets `auto_quarantine_enabled` (with `auto_quarantine_min_flips` / `auto_quarantine_min_runs` on `organizations`, read via `pool.query` — that table has no RLS), `evaluateAutoQuarantine(orgId, suite)` runs at run finalization (`routes/runs.ts`, **awaited before `postPRComment`** so the soften-check sees fresh entries) and quarantines any test whose `flip_count`/`total_runs` clear the policy, with `source='auto'`. The insert is `ON CONFLICT DO NOTHING` — a human's manual quarantine keeps its reason/owner; only brand-new flaky tests get an auto row. It's defensive: any failure logs and returns 0 (finalization must never break on this side-effect).
- **DELIBERATE soften-check asymmetry — the external git gate relaxes, the dashboard/badge stay honest.** When a run has failures but *every* failed test is quarantined for that suite (`computeAllFailuresQuarantined` in `git-providers/index.ts`), `postPRComment` posts a **success/neutral** commit status / Checks conclusion so a known-flaky-only failure doesn't block the PR. That relaxation is scoped to the *external merge gate only*: `classifyRunStatus()`, the SVG badge, and `GET /runs/status` keep reporting `failed`. Flakey's own dashboard never lies — quarantine just stops a known-flaky test from blocking a merge. The check **fails closed**: an empty failed-titles list (e.g. `stats.failed > 0` with no `status==='failed'` rows) returns false, and a failed quarantine lookup falls back to the honest failure state.
- **`flaky.threshold.exceeded` webhook + org thresholds.** Beyond `flaky.detected`, dispatch fires `flaky.threshold.exceeded` when a test's `flaky_rate` crosses the org's `flaky_alert_threshold` (`organizations`, no RLS — read via `pool.query` on the trusted orgId, the `getProviderConfig` pattern). A NULL column means alerting is off (skip). Candidates need `total_runs >= 3`; the payload carries up to 10 over-threshold tests. Guarded so a failed config read logs and doesn't abort the rest of dispatch. `GET /webhooks/events` is the single source of truth for selectable events (derived from `VALID_EVENTS`) — don't hardcode the event list anywhere else.
- **AI fix-PR (`POST /analyze/fix-pr`) needs a git token with WRITE / PR scope — an elevated, opt-in scope.** The PR-comment / commit-status / Checks tokens in use today may be read-only; opening a draft fix PR additionally needs branch-create + file-commit + PR-create permission (GitHub: `contents:write` + `pull_requests:write`; GitLab: `api` / write_repository + MR scope; Bitbucket: repo write + PR). The route is **draft-only, never auto-merged**, bounded by a 40 000-char file cap (413), guarded against empty/unchanged (no PR) and truncated (< half size → 422) output, and idempotent per `(org, target_type, target_key)` via the `ai_fix_prs` table (migration 061). Contributor+ only. See `docs/integrations.md`.
- **AI is air-gapped-capable and instance-wide.** All AI routes (`analyzeFailure` / `analyzeFlakyTest` / `analyzeCluster` / `generateFixPatch`, via `chatJSON` in `ai.ts`) call the instance-configured provider. Set `AI_PROVIDER=openai` + `AI_BASE_URL=http://localhost:11434/v1` to run everything against a **local Ollama** with no data egress. `isAIEnabled()` gates generation (503 when unset); cached results still read with AI off, and clustering's similarity grouping is cost-free either way.
