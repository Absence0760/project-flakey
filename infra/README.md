# Flakey — AWS Infrastructure

Terraform configuration for deploying Flakey to AWS. Designed to be as touchless as possible — secrets are auto-generated, migrations run automatically on deploy, and monitoring is built in.

## Architecture

```
                    Internet
                       |
              +--------+--------+
              |                 |
        CloudFront (HTTPS)   ALB (HTTP)
              |                 |
        S3 (frontend)    ECS Fargate (backend)
        static files     [auto-scales 1-4]
                              |
                   +----------+----------+
                   |                     |
             RDS PostgreSQL        S3 (artifacts)
             [encrypted,          [screenshots,
              7-day backups]       videos, snapshots]
```

## What's automated

| Concern | How it's handled |
|---|---|
| Secrets (DB password, JWT) | Auto-generated by Terraform, stored in AWS Secrets Manager |
| Database migrations | Run automatically on container startup via `entrypoint.sh` |
| App role (`flakey_app`) | Created by migration 005, runs on first deploy |
| RLS policies | Applied by migrations, idempotent |
| Auto-scaling | CPU-based, 1-4 tasks, scale out at 70% CPU |
| Health monitoring | ALB health check verifies DB connectivity, CloudWatch alarms for unhealthy hosts and 5xx errors |
| Frontend deploy | Static build synced to S3, CloudFront cache invalidated |
| GitHub OIDC | Bootstrap module creates the IAM role |

## Self-hoster checklist

**Nothing in this stack hard-codes `flakey.io`** — every domain, cert, region, and IAM-trust-subject is exposed as a variable. To run this for your own org:

| Variable | What it controls |
|---|---|
| `app_name` | Prefix for every named resource (ECR repo, ECS cluster + service, log groups, security groups). Two `app_name`s never collide in one AWS account. |
| `aws_region` | Where the ALB / ECS / RDS live. CloudFront cert must additionally be in `us-east-1` (an AWS hard constraint). |
| `acm_certificate_arn` | Your ALB cert. Must be in `aws_region`. |
| `cloudfront_acm_certificate_arn` | Your CloudFront cert. Must be in `us-east-1`. |
| `cloudfront_aliases` | Your DNS names (e.g. `["app.acme.com"]`). |
| `csp_connect_src` | Your API origin(s) so the SPA can fetch the API across origins. **Required** — `terraform apply` refuses if absent or still holding the `<placeholder>` string from `terraform.tfvars.example`. |
| `allow_registration` | `true` for SaaS-style self-serve; `false` (default) for invite-only. |
| `budget_alert_email` | Where the monthly-budget alarm sends mail. |
| `github_repo` (bootstrap stack only) | Your `<org>/<repo>` for the OIDC trust subject. The deploy IAM role's trust policy is scoped to this — no other repo can assume it. |

Copy `terraform.tfvars.example` → `terraform.tfvars`, fill in each `<placeholder>`, then `terraform apply`. The validation blocks in `variables.tf` will refuse to plan if you leave a placeholder in place.

The reporter packages (`@flakeytesting/*`) and the backend itself read `FLAKEY_API_URL` / `FLAKEY_API_KEY` at runtime — there are no compile-time references to `flakey.io` anywhere in `backend/` or `packages/`.

## Setup (one-time)

### 1. Bootstrap (state bucket + GitHub OIDC)

```bash
cd infra/bootstrap
terraform init
# Replace <your-github-org>/<your-repo> with your fork's path. The
# deploy IAM role's trust policy is StringLike-pinned to this exact
# value — a fork that never set this won't be assumable.
terraform apply -var="github_repo=<your-github-org>/<your-repo>"
```

This creates:
- S3 bucket for Terraform state
- DynamoDB table for state locking
- GitHub OIDC provider + IAM role for GitHub Actions

Save the output `github_actions_role_arn` — you'll need it for GitHub secrets.

### 2. Deploy infrastructure

```bash
cd infra
terraform init
terraform plan
terraform apply
```

No passwords to configure — they're auto-generated and stored in Secrets Manager.

### 3. Set up GitHub secrets

Add these to your GitHub repository (Settings > Secrets > Actions):

| Secret | Value | How to get it |
|---|---|---|
| `AWS_ROLE_ARN` | IAM role ARN | `cd bootstrap && terraform output github_actions_role_arn` |
| `API_URL` | Backend URL | `terraform output alb_dns_name` (prefix with `http://`) |
| `FRONTEND_BUCKET` | S3 bucket name | `terraform output frontend_bucket` |
| `CLOUDFRONT_DISTRIBUTION_ID` | CloudFront ID | `terraform output cloudfront_distribution_id` |
| `NPM_TOKEN` | npm access token | https://www.npmjs.com/settings/tokens (only if publishing packages) |

### 4. Push to deploy

```bash
git push origin main
```

GitHub Actions will:
1. Build Docker image → push to ECR
2. Deploy to ECS → container starts → migrations run automatically → app starts
3. ALB health check verifies DB connectivity before routing traffic
4. Build frontend → sync to S3 → invalidate CloudFront

### 5. Access the app

```bash
terraform output frontend_url    # Frontend
terraform output alb_dns_name    # Backend API
```

Register the first user (or seed data via a bastion connection).

## Deploy flow (after initial setup)

```
Push to main
    |
    +-- backend/** changed?
    |     Build Docker image
    |     Push to ECR
    |     ECS force-new-deployment
    |     Container starts:
    |       1. Run migrations (as superuser)
    |       2. Start app (as flakey_app)
    |     ALB health check (GET /health, checks DB)
    |     Old task drains
    |
    +-- frontend/** changed?
          pnpm build (static files)
          aws s3 sync to bucket
          CloudFront cache invalidation
```

Zero manual steps after initial setup.

## CI/CD Pipelines

| Workflow | Triggers on | What it does |
|---|---|---|
| `deploy.yml` | `backend/**` or `frontend/**` changes | Backend: Docker → ECR → ECS. Frontend: build → S3 → CloudFront |
| `publish.yml` | `packages/**` changes | Publishes npm packages (`@flakeytesting/cli`, `@flakeytesting/core`, `@flakeytesting/cypress-reporter`, `@flakeytesting/playwright-reporter`, `@flakeytesting/webdriverio-reporter`, `@flakeytesting/cypress-snapshots`, `@flakeytesting/playwright-snapshots`) |

## Monitoring

- **Health check**: `GET /health` verifies database connectivity (returns 503 if DB is down)
- **CloudWatch alarms**:
  - Unhealthy host count > 0 → SNS alert
  - 5xx errors > 10 in 5 min → SNS alert
- **Container Insights**: CPU, memory, network metrics in CloudWatch
- **CloudWatch Logs**: Backend logs retained 30 days

Subscribe to the SNS topic `flakey-production-alerts` for email/Slack notifications.

## Security

- Secrets stored in AWS Secrets Manager (not plaintext env vars)
- ECS execution role has minimal permissions (pull images + read secrets)
- ECS task role only has S3 access for the artifacts bucket
- RDS in private subnets, only accessible from ECS security group
- S3 artifacts bucket blocks all public access
- RDS encrypted at rest, final snapshot on destroy
- GitHub Actions uses OIDC (no stored AWS credentials)

## Estimated Cost

| Resource | Monthly Cost |
|---|---|
| RDS db.t4g.micro | ~$15 |
| ECS Fargate (1 task) | ~$10 |
| NAT Gateway | ~$30 |
| ALB | ~$16 |
| S3 + CloudFront | ~$1 |
| Secrets Manager | ~$1 |
| **Total** | **~$73/month** |

### Reducing costs

- Replace NAT Gateway with a NAT Instance (~$4/month, saves ~$26)
- Single EC2 instance with Docker Compose + Caddy (~$10/month total)

| `budget` | Monthly AWS budget with 80%/100%/forecasted email alerts |

## Adding a custom domain

1. Register or transfer your domain to Route53
2. Request an ACM certificate (us-east-1 for CloudFront, your region for ALB)
3. Add the certificate to the CloudFront distribution and ALB listener
4. Create Route53 alias records pointing to CloudFront (frontend) and ALB (API)
5. Update `CORS_ORIGINS` in the ECS task definition to your domain
